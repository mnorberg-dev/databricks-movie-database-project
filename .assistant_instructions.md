# Assistant Instructions

## Python Guidelines

### Code Structure & Style
* Use Google-style docstrings for all functions, classes, and modules
* Add descriptive comments before each logical section to explain intent
* Follow PEP 8 naming conventions:
  * `snake_case` for functions and variables
  * `PascalCase` for classes
  * `UPPER_CASE` for constants
* Keep functions focused and single-purpose
* Use type hints on all function arguments and return values for clarity and IDE support
* Use the integrated formatter in Databricks for consistent code formatting
* Blank lines should be truly blank (no trailing spaces or tabs)
* Place comments on their own line before the statement they describe, not as inline comments at the end of lines
* Prefer multi-line comment blocks at the top of a code section with a blank line before the code:
  ```python
  # Good - comment block at top with blank line
  # Retrieve API key and initialize client
  # This ensures secure credential handling

  api_key = dbutils.secrets.get(scope="tmdb", key="api_key")
  client = MovieAPIClient(api_key=api_key)
  
  # Bad - inline comments at end of lines
  api_key = dbutils.secrets.get(scope="tmdb", key="api_key")  # Get API key
  client = MovieAPIClient(api_key=api_key)  # Initialize client
  ```

### Imports
* Follow PEP 8 import organization:
  * Standard library imports (sorted alphabetically)
  * Third-party imports (sorted alphabetically)
  * Local application imports (sorted alphabetically)
  * Separate each group with a blank line
* When using multi-line imports (parentheses), sort all imported names alphabetically:
  ```python
  # Good
  from pyspark.sql.types import (
      ArrayType,
      FloatType,
      IntegerType,
      StringType,
      StructField,
      StructType,
  )
  
  # Bad - not sorted
  from pyspark.sql.types import (
      StructType,
      StructField,
      StringType,
      IntegerType,
      FloatType,
      ArrayType,
  )
  ```

### Error Handling
* Use specific exception types rather than bare `except:`
* Validate inputs early in functions
* Provide informative, actionable error messages that help debug issues
* Log errors with context (what operation failed, relevant values)
* Include troubleshooting hints in error messages when possible

### Best Practices
* Follow DRY (Don't Repeat Yourself) - extract repeated logic into functions
* Prefer explicit over implicit (clear code over clever code)
* Use list/dict comprehensions for simple transformations, loops for complex logic
* Avoid deeply nested code (max 3-4 levels)
* Define constants at the top of the file or in a separate config module
* Avoid magic numbers and strings - use named constants
* When using Spark, prefer DataFrame API over RDDs for performance
* Use f-strings for string formatting (more readable than % or .format())

### Testing
* Write unit tests for all business logic functions
* Test edge cases: empty inputs, None values, boundary conditions
* Use descriptive test names that explain what is being tested
* Mock external dependencies (APIs, databases) in unit tests
* Include example inputs and expected outputs in test docstrings

## SQL Guidelines

### Query Organization
* Break complex tasks into logical steps using temporary views
* Each temp view should represent one clear transformation
* Avoid queries that do too many things at once

### Style Standards
* CAPITALIZE SQL keywords (SELECT, FROM, WHERE, JOIN, etc.)
* Use explicit JOIN syntax (INNER JOIN, LEFT JOIN) - never implicit joins
* Always use table aliases for multi-table queries
* Use meaningful alias names (not just `a`, `b`, `c`)
* Indent subqueries and JOIN conditions for readability
* Use the integrated formatter in Databricks for consistent code formatting
* **Clause Formatting**: Each major SQL clause (SELECT, FROM, WHERE, GROUP BY, ORDER BY, LIMIT) should be on its own line:
  ```sql
  -- Good - each clause on its own line
  SELECT
    column1,
    column2,
    column3
  FROM
    table_name
  WHERE
    condition = TRUE
  ORDER BY
    column1 DESC
  LIMIT 10;
  
  -- Bad - single line
  SELECT column1, column2 FROM table_name WHERE condition = TRUE;
  ```
* Move inline comments to their own lines before the query, not at the end of WHERE clauses or other statements

### Comments
* Add a comment block at the top explaining the query's purpose
* Comment each major section or CTE
* Explain any complex business logic or non-obvious filtering

### Unity Catalog Best Practices
* Always add COMMENT on tables and views describing their purpose and content
* Add column comments for non-obvious fields, especially business metrics
* Include information about data refresh frequency in table comments
* Document data lineage in comments (where the data comes from)
* See the Unity Catalog Documentation skill for detailed step-by-step procedures

## Data Pipeline Architecture

### Notebook Organization
* **Prefer split architectures**: Separate notebooks for each layer/stage rather than monolithic pipelines
* **Naming convention**: Use numeric prefixes to indicate execution order
  * `01-bronze-ingestion` (or `01-ingest-raw-data`)
  * `02-silver-transformations` (or `02-clean-and-normalize`)
  * `03-gold-aggregations` (or `03-create-aggregates`)
  * `04-analysis-queries` (or `04-ad-hoc-analysis`)
* **Language choice**:
  * Bronze ingestion: Python (for API calls, file processing, complex logic)
  * Silver/Gold transformations: SQL (for data transformations and aggregations)
  * Analysis: SQL (for queries and reporting)
* **Benefits of split architecture**:
  * Clear separation of concerns
  * Easier debugging (isolate failures to specific stages)
  * Independent testing and development
  * Better version control (smaller diffs, clearer changes)
  * Reusability (downstream notebooks can be reused with different sources)
  * Parallel development (team members can work on different stages)

### Medallion Architecture Pattern
* **Bronze layer**: Raw data ingestion, minimal transformation
  * Store data as close to source format as possible
  * Include ingestion metadata (timestamp, source, batch ID)
  * Handle schema evolution and data quality issues at source
* **Silver layer**: Cleaned and conformed data
  * Standardize column names and data types
  * Apply business rules and data quality checks
  * Deduplicate and merge records
  * Add derived fields (calculated columns, enrichments)
* **Gold layer**: Aggregated and business-ready data
  * Pre-computed aggregations and metrics
  * Denormalized for query performance
  * Business-friendly naming and structure
  * Optimized for specific use cases (dashboards, reports, ML features)

### Pipeline Dependencies
* Document execution order clearly (via naming convention and README)
* Each notebook should verify upstream dependencies exist before running
* Include table/view existence checks at the start of downstream notebooks
* Consider using Databricks Workflows or Delta Live Tables for production orchestration
* For complex dependencies, create a DAG diagram in the project README

### Example Project Structure

## General Code Quality

### Configuration Management
* Store configuration in separate files (config.py, config.json, .env)
* Never commit secrets or credentials to version control
* Use environment-specific configs (dev, staging, prod)
* Document all configuration parameters with defaults

### Code Reusability
* Extract repeated code into reusable functions
* Create utility modules for common operations
* Build libraries for team-wide patterns
* Share common SQL views and tables across projects

### Performance Considerations
* Profile before optimizing - measure don't guess
* For Spark: avoid `.collect()` on large datasets, use broadcast joins when appropriate
* For SQL: check query plans for large tables, consider partitioning strategies
* Cache intermediate results when they're reused multiple times
* Use appropriate data types (INT vs BIGINT, VARCHAR vs TEXT)

## Documentation & Maintenance

### README Files
* After completing major tasks in a git repository, review the README
* Remove stale sections that no longer reflect the codebase
* Add new sections for features, setup steps, or architecture changes
* Keep dependencies, prerequisites, and usage examples up to date

### In-Code Documentation
* Update comments when code logic changes
* Document "why" not "what" - the code shows what, comments explain why
* Add TODO/FIXME comments for known issues or future improvements
* Include links to relevant Jira tickets, docs, or PRs for context

### Notebooks
* Use markdown cells to create a narrative flow
* Optionally include a header cell with purpose, author, and last updated date (recommended for complex notebooks, may be overkill if sufficiently documented in README)
* Document assumptions about data (freshness, schema, filters applied)
* Show sample outputs to verify transformations are correct
* Clear all output before committing notebooks to version control
* **Naming Convention**: Use dashes instead of spaces in notebook names (e.g., `customer-analysis` not `Customer Analysis`)
* **Format Preference**: Use source format (.py files) rather than Jupyter format (.ipynb) when working with GitHub for better diff and merge capabilities


### Version Control
* Write clear, descriptive commit messages in imperative mood ("Add feature" not "Added feature")
* Keep commits focused on a single logical change
* Update .gitignore to exclude generated files, credentials, and local config
* Review diffs before committing to catch unintended changes

## Security & Data Privacy

* Never hardcode credentials, tokens, or sensitive values
* Use Databricks secrets or environment variables for sensitive config
* Be cautious with PII - avoid printing, logging, or sampling PII columns unnecessarily
* When working with sensitive data, confirm appropriate access controls are in place
* Sanitize user inputs to prevent SQL injection
* Use parameterized queries instead of string concatenation for dynamic SQL
* Redact sensitive data in logs and error messages
